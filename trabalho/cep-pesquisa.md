# CEP Lookup Skill
Captura, padroniza e organiza todas as informações possíveis extraídas a partir de um CEP brasileiro (Código de Endereçamento Postal), utilizando como base os dados oficiais dos Correios e fontes complementares (IBGE).
## When to Use This Skill
- Usuário informa um CEP e deseja obter o endereço completo- Necessidade de validar ou autocompletar campos de endereço- Popular registros de banco de dados com dados padronizados de localização- Capturar coordenadas geográficas a partir de um CEP- Enriquecer cadastros com informações regionais (DDD, código IBGE, região, fuso horário)- Triggers: "consultar CEP", "buscar CEP", "endereço do CEP", "dados do CEP", "informações do CEP", "extrair CEP"
## How to Use
### Step 1: Receber e Validar o CEP
- O CEP deve conter **8 dígitos numéricos**- Formatos aceitos: `21371020`, `21371-020` ou `21.371-020`- Normalizar para o formato padrão com hífen: `XXXXX-XXX` (ex: `21371-020`)- Se o CEP for inválido ou incompleto, solicitar correção ao usuário
### Step 2: Consultar os Dados
Realizar busca utilizando fontes confiáveis:
1. **Fonte primária:** API dos Correios / ViaCEP (`viacep.com.br/ws/{cep}/json/`)2. **Fonte secundária:** BrasilAPI (`brasilapi.com.br/api/cep/v2/{cep}`)3. **Fonte complementar:** Web Search para dados adicionais (coordenadas, DDD, região)
### Step 3: Extrair e Padronizar os Campos
Organizar os dados extraídos nos seguintes campos padronizados:
| # | Campo | Descrição | Exemplo ||---|-------|-----------|---------|| 1 | `cep` | CEP formatado (XXXXX-XXX) | `21371-020` || 2 | `cep_numerico` | CEP somente números (8 dígitos) | `21371020` || 3 | `tipo_logradouro` | Tipo do logradouro (Rua, Avenida, Travessa, etc.) | `Rua` || 4 | `logradouro` | Nome do logradouro (sem o tipo) | `César Múzio` || 5 | `logradouro_completo` | Tipo + Nome do logradouro | `Rua César Múzio` || 6 | `complemento` | Complemento do CEP (se houver) | `até 599/600` || 7 | `bairro` | Nome do bairro/distrito | `Vicente de Carvalho` || 8 | `cidade` | Nome da cidade/município | `Rio de Janeiro` || 9 | `uf` | Sigla do estado (2 letras) | `RJ` || 10 | `estado` | Nome completo do estado | `Rio de Janeiro` || 11 | `regiao` | Região do Brasil | `Sudeste` || 12 | `pais` | País | `Brasil` || 13 | `ddd` | Código DDD da localidade | `21` || 14 | `codigo_ibge` | Código IBGE do município (7 dígitos) | `3304557` || 15 | `latitude` | Latitude (decimal, referência IBGE) | `-22.85416` || 16 | `longitude` | Longitude (decimal, referência IBGE) | `-43.31694` || 17 | `fuso_horario` | Fuso horário IANA | `America/Sao_Paulo` || 18 | `cep_ativo` | Status do CEP (ativo/inativo) | `true` |

Best Practices
•	Sempre normalizar o CEP para o formato XXXXX-XXX antes de consultar
•	Nunca assumir dados — se um campo não for retornado pela API, marcar como null
•	Separar tipo e nome do logradouro (ex: "Rua" e "César Múzio") para facilitar buscas e filtros no banco
•	Usar coordenadas do IBGE como referência — são aproximações do centroide do município, não do endereço exato
•	Validar o DDD — nem todas as APIs retornam esse campo; pode ser necessário consulta complementar
•	Tratar CEPs especiais — CEPs de caixas postais, grandes recebedores e unidades dos Correios podem não ter logradouro
•	Cache de consultas — sugerir ao usuário implementar cache para evitar chamadas repetidas à API

________________________________________
📮 Por que dois campos para o CEP?
Campo cep — CHAR(9)
É o CEP com a máscara/formatação, incluindo o hífen.
21371-020│││││ │││12345 6789  →  9 caracteres
Uso: Exibição em telas, relatórios, etiquetas de envio, integração com Correios.
________________________________________
Campo cep_numerico — CHAR(8)
É o CEP sem formatação, apenas os dígitos.
21371020││││││││12345678  →  8 caracteres
Uso: Buscas, filtros, chamadas de API, comparações e indexação no banco.
________________________________________
🤔 Mas por que não guardar só um?
Cenário	Qual campo usar?
Exibir no front-end / PDF / etiqueta	cep (formatado)
Chamar API do ViaCEP / BrasilAPI	cep_numerico (sem hífen)
Busca/filtro no banco de dados	cep_numerico (mais performático)
Validação de entrada do usuário	cep_numerico (comparar só números)
Enviar para os Correios	cep (formatado)
Se guardar só o formatado, toda vez que precisar chamar uma API ou fazer uma busca, teria que remover o hífen em tempo de execução.
Se guardar só o numérico, toda vez que precisar exibir na tela, teria que formatar adicionando o hífen.
💡 Guardando os dois, você elimina processamento repetitivo e simplifica tanto o back-end quanto o front-end.
________________________________________
⚡ Na prática
sql
-- Busca rápida por CEP (usa o campo sem máscara)SELECT * FROM ceps WHERE cep_numerico = '21371020';
-- Exibição para o usuário (já vem formatado, sem tratar)SELECT cep, logradouro_completo, bairro, cidade, uf FROM ceps WHERE id = 1;-- Resultado: 21371-020 | Rua César Múzio | Vicente de Carvalho | Rio de Janeiro | RJ
________________________________________
🔄 Alternativa: guardar só um campo
Se preferir simplificar a tabela, você pode manter apenas o cep_numerico (CHAR(8)) e formatar no código da aplicação:
python
# Python — formatar na hora de exibircep_formatado = f"{cep_numerico[:5]}-{cep_numerico[5:]}"# "21371020" → "21371-020"
javascript
// JavaScript — formatar na hora de exibirconst cepFormatado = `${cep.slice(0, 5)}-${cep.slice(5)}`;// "21371020" → "21371-020"
Nesse caso, a tabela ficaria com 1 campo a menos e a formatação ficaria por conta da aplicação.
________________________________________
✅ Recomendação
Abordagem	Quando usar
Dois campos (cep + cep_numerico)	Sistemas com múltiplos consumidores (APIs, relatórios, front-end) — evita reprocessamento
Um campo (só cep_numerico)	Sistemas menores/simples onde a formatação no código não é problema

⚠️ Observações Importantes sobre os Nomes das Colunas
Percebi que você usou hífen (-) nos nomes das colunas (cep-id, cep-bairro). No SQL, o hífen é interpretado como operador de subtração. Use underscore (_) no lugar:
❌ Não usar	✅ Usar
cep-id	cep_id
cep-bairro	cep_bairro
cep-info-localizacao	cep_info_localizacao
cep-metroquadrado-loft	cep_m2_loft
cep-info-custo-benefício	cep_info_custo_beneficio (sem acento!)
